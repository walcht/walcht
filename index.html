Virtual Exhibition

TECHNOLOGIES: Unity3D, C#, Git LFS, Blender, GIMP, JavaScript

TARGET PLATFORM(s): WebGL

Virtual Exhibtion is a highly-customizable, immersive, 3D job fair that runs flawlessly on WebGl. This is a freelance software development project done by a team of 2 people: fullstack web developer and game developer (me).

Since I wanted to learn as much as possible from this project, only two assets from the Unity Store were used (for 3D Characters and sounds), everything else was built\done from scratch; Character Movement, AI System, 3D Models, Camera Collision Avoidance, Lighting, Animations, etc...
Since the target platform for this project is WebGl, we were confronted with numerous challenges, many of which we have managed to solve. Below are a list of the most difficult challenges I've faced.

Challenges

  1. WebGl is single-threaded. CPU optimization is extremely important
  2. Highly coupled and unmaintainable code base
  3. Animations took alot of CPU time
  4. Customizable Textures have to be requested during runtime
  5. High numbers of draw calls and high number of vertices
  6. Indoor realtime lighting heavily impacted game's performance

Solutions

  1. Reduced the number of materials by sharing materials across numerous 3D models. Heavy use of static batching to reduce state changes.
  2. ScriptableObjects as intermediate layers; a ScriptableObject was used as an intermediate layer between the Input Manager and the objects that required input. Another ScriptableObject was also used as a Scene Management intermediate layer. I've also relied heavily on using Manager Singletons. For instance, we had a UIManager, CustomizationManager, TextureRequestManager, etc... All of these were put into a seperate, persistent Scene that holds all persistent GameObjects. This has greatly improved the structure of the project.
  3. Use environment to properly set up Occlusion Culling to completely cull NPC characters and set a maximum distance after which any character won't be rendered.
  4. I built a TextureRequestManager that receives texture requests and executes them when possible. Since, for each scene, we have hundreds of texture requests, this manager solved alot of issues by limiting how many texture requests can be served at a time.
  5. Manually merge models that share the same material together, I did this for all the roof 3D models.
  6. Lightmapping resulted in better visuals with much better performance. Using light probes runtime-instantiated objects resulted in good visuals as well.

We've learned that

  • Modern principles of software development are extremely important. Not following these results in an unmaintainable project.
  • Singletons come with alot of disadvantages in Unity development environment. Using ScriptableObjects when possible proved to be a much better design choice.
  • We should optimize for the target platform. WebGl is single-threaded, most performance issues are related to CPU-bottleneck. For example, reducing vertex-count of 3D objects won't help that much.
  • Using a CI\CD pipeline could've saved us alot of time. Countless times we had to rebuild the whole project because of an error that could had been easily detected.
  • Using the Unity Profiler in the development build helped us identifying performance issues.
